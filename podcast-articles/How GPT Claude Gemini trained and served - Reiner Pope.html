一、Batch Size：延迟与成本的秘密

讲座从一个简单的问题开始：为什么 Claude Code、Cursor 等工具提供「Fast Mode」——6 倍价格换来 2.5 倍速度？背后的机制是什么？

Reiner 给出的答案非常直接：Batch Size（批大小）是推理效率的主要杠杆。

Batch Size = 1

高延迟、低吞吐量。每个用户请求独立处理，权重加载成本无法分摊。

成本 ≈ 权重参数总量 ÷ 内存带宽

Batch Size = 100+

低延迟、高吞吐量。权重加载成本被大量请求分摊，计算与内存达到平衡。

最优成本 ≈ 活跃参数 ÷ GPU 计算性能

关键洞察：如果不将多个用户请求批量处理，经济性会比批处理差 1000 倍。

「Fast Mode 和 Slow Mode 是延迟-吞吐量权衡光谱上的不同位置。支付更多钱，你得到的是专用资源，允许更小的有效 Batch Size，从而降低延迟。」 — Reiner Pope

二、Roofline 模型：理解 GPU 性能

Reiner 使用经典的 Roofline 分析框架来量化推理性能。核心是两个时间约束：

时间 = max(计算时间, 内存时间) 计算时间 = (活跃参数 × Batch Size) ÷ GPU 峰值算力 内存时间 = (总参数 + Batch Size × 上下文长度 × 每Token字节数) ÷ 内存带宽

对于一个 1 万亿参数模型（如 GPT-4 级别）：

  • Batch Size = 1：内存带宽受限（Memory Bound），总内存 4TB，内存时间约 4 秒，计算时间 2 秒
  • Batch Size = 100：计算与内存达到平衡，总内存 200TB，两种时间都是 200 秒
关键公式：

最优 Batch Size ≈ 300 × 稀疏度

其中「300」是硬件常数（FLOPs ÷ 内存带宽），稀疏度 = 活跃参数 ÷ 总参数。

对于 DeepSeek V3（激活 32/256 个专家），稀疏度 = 8，最优 Batch Size ≈ 2400。

这个「最优 Batch Size」只取决于稀疏度，与模型大小无关——这是一个非常有趣的结论。

三、KV Cache：被忽视的内存大户

Key-Value Cache 是 Attention 机制中存储历史 Token 内部表示的内存区域。Reiner 强调了一个反直觉的事实：

KV Cache 的内存占用 ≈ 模型权重本身

对于 Batch Size = 1，KV Cache 大小与权重相当；对于更大 Batch Size，KV Cache 成为主导。

这意味着：

  • KV Cache 无法通过增加 Batch Size 来分摊（每个请求的 KV Cache 都是独立的）
  • KV Cache 无法跨流水线阶段分片（Pipeline Parallelism 无效）
  • 长上下文 = 更多 KV Cache = 更贵的内存

四、MoE 模型：如何跨 GPU 分布

稀疏混合专家（MoE）模型将不同专家放置在不同 GPU 上，使用「专家并行」（Expert Parallelism）。以 DeepSeek V3 为例：

  • 256 个专家，分布在 64 GPU（每 GPU 4 个专家）
  • 每个 Token 路由到约 32 个专家
  • 通信模式：All-to-All（全连接）

关键问题：为什么一个机架（Rack）限制了专家层的规模？

答案是 NVLink vs. Scale-out 网络的带宽差异。Nvidia Blackwell 机架内使用 NVLink（快速互联），跨机架使用慢 8 倍的 Scale-out 网络。当 Token 需要跨机架路由时，就成为瓶颈。

物理约束

机架大小受限于：空间、重量、供电、冷却。当 GPU 数量增加，线缆密度成为关键瓶颈——「弯曲半径」等物理限制使得线缆不能无限密集。

从 Hopper（8 GPU）到 Blackwell（72 GPU）到 Rubin（500+ GPU）的演进，主要是应对这些物理工程挑战。

五、流水线并行：跨机架的权衡

除了专家并行，还有两种并行策略：

  • 数据并行：数据分片
  • 流水线并行：模型层分片到不同机架

Reiner 指出，流水线并行在推理中的效果是中性——它不改善也不恶化延迟，只是减少了每机架的内存占用。但在训练中，它面临「Pipeline Bubble」问题。

「流水线并行在推理中是一种无需动脑的选择，但实际使用的不多，因为 Blackwell 机架有几十 TB 内存，远超 1 万亿参数模型所需的 1TB。」 — Reiner Pope

最终结论：在推理中，主要使用专家并行 + 少量流水线并行，因为 KV Cache 无法跨阶段分片。

六、Scaling Laws：为什么模型过度训练 100 倍

Reiner 提出了一个关于 RL 时代训练 Scaling 的有趣分析：

成本分解

  • 预训练成本 ≈ 6 × 活跃参数 × 数据量
  • RL 成本 ≈ 2-6 × 活跃参数 × RL 数据量
  • 推理成本 ≈ 2 × 活跃参数 × 推理数据量

最优分配

最小化总成本 ≈ 预训练成本 + RL 成本 + 推理成本

经验法则：三种成本应大致相等

基于这个框架和实际数据推断：

  • 全球 AI 推理流量：约 5 亿 Token/秒
  • 模型部署周期：约 2 个月
  • 推理 Token 总量：约 200 万亿
  • 预训练 Token 总量（GPT-5 级别）：约 100-200 万亿

核心洞察

当前前沿模型比 Chinchilla 最优训练量过度训练约 100 倍。这是因为 RL 和推理的成本占据了与预训练相当的比例。

七、长上下文的成本：从 API 定价推断

Gemini 3.1 的定价策略揭示了底层成本结构：超过 200K Token 的上下文贵 50%。

Reiner 演示了如何从定价推断技术细节：

  • 200K Token 是「甜蜜点」——内存时间与计算时间交叉的位置
  • 推断每 Token 约 2KB KV Cache（128 维头 × 8 KV 头）

另一个洞察：Prefill（输入处理）比 Decode（输出生成）便宜 5 倍。这是因为 Prefill 是计算受限（Compute Bound），而 Decode 是内存带宽受限（Memory Bound）。

「从 API 定价，你可以反推他们的成本结构。如果 API 提供商定价接近成本，否则别人就会抢占市场。」 — Reiner Pope

八、多级内存：HBM、DDR、Flash 的经济学

不同内存层级的成本-延迟权衡：

HBM（高速带宽内存）

  • 成本：按占用比例计费
  • 用途：短期 KV Cache（5 分钟）

DDR / 主机内存

  • 成本：比 HBM 低但带宽也低
  • 用途：中期缓存

Flash / NVMe

  • 成本：最低但延迟最高
  • 用途：长期存储 / 重新计算

决策规则：选择使「检索成本 + 持有成本」最小化的内存层级。

核心要点总结

  1. Batch Size 是效率杠杆：最优值 ≈ 300 × 稀疏度，决定了计算与内存的平衡
  2. KV Cache 无法分摊：不能跨 Batch 共享，不能跨 Pipeline 分片
  3. 稀疏度决定最优规模：与模型大小无关，只取决于活跃/总参数比
  4. 机架是物理边界：线缆密度限制了单 rack 的 GPU 数量
  5. 模型过度训练 100 倍：因为 RL 和推理成本占据与预训练相当的比例
  6. 上下文长度受限于内存带宽：200K Token 是当前的甜蜜点
  7. API 定价泄露成本结构：可以反推技术实现细节
「理解训练和推理在集群中如何工作，许多事情——为什么 AI 是现在的样子，为什么 AI 架构是现在的样子，为什么 API 定价是现在的样子——都开始变得有意义。」 — Dwarkesh Patel（主持人）
